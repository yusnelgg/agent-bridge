AgentBridge

Conecta agentes de IA entre distintas PCs para que colaboren y compartan información en tiempo real.

v0.2.0 • Go + NATS + WebSocket + MCP

Introducción

Agent Bridge es un daemon en Go que permite que agentes de IA (opencode, Claude Code, Cursor, cualquier AI que pueda ejecutar comandos) se comuniquen entre sí a través de distintas PCs en tiempo real. Es agnóstico al modelo de IA: cada PC puede usar el asistente que quiera.

Caso de uso típico: Dos desarrolladores — uno frontend, uno backend — en distintas PCs o países. El frontend le pide a su IA un endpoint, su IA le pregunta a la IA del backend via agent-bridge ask --wait, la IA del backend lo implementa y responde con instrucciones de consumo. El frontend solo implementa la UI.

Características

  • Sin dependencias — Un solo binario, no necesitas instalar nada más
  • Tiempo real — WebSocket push, sin polling. Respuesta instantánea
  • IA Agnóstico — Cada lado usa el AI que quiera: opencode, Claude Code, Cursor, etc.
  • Rol definido — Frontend solo consume (UI), Backend solo programa (endpoints, lógica)
  • MCP Server — Claude Code y Cursor se conectan nativamente via MCP
  • Persistencia SQLite — Mensajes y tareas se guardan localmente
  • Multi-PC — Conexión via Tailscale, ZeroTier, o VPS
  • Embedded NATS — Message broker embebido, 0 config

Cada bridge expone una API HTTP + WebSocket. Cuando el frontend envía un mensaje via agent-bridge ask --wait, el bridge lo publica en NATS. El bridge backend lo recibe, lo persiste, y lo envía al cliente via WebSocket push. Si WebSocket no está disponible, cae automáticamente a polling (2s).

REGLA DE ORO

Separación estricta de responsabilidades

FRONTEND → SOLO consume. Pide, recibe respuesta, implementa la UI.
⚡
BACKEND → SOLO programa. Recibe pedidos, escribe TODO el backend, responde con instrucciones de consumo.

Ninguno programa lo que le corresponde al otro.

Frontend

Pedís algo, el backend programa todo, te dice cómo consumirlo.

agent-bridge ask --wait backend "Endpoint GET /api/users que devuelva lista de usuarios con id, nombre, email. Dame el código completo del endpoint, modelo y cómo consumirlo."

El backend responde con:

Express con endpoint en /api/users. Modelo User con id, nombre, email.
Probá con: curl http://localhost:3000/api/users
📦 Código completo en server.js línea 10-45

Vos solo implementás la UI para consumirlo.

Backend

Escuchás pedidos, programás TODO, respondés con instrucciones claras.

AGENT_BRIDGE=http://localhost:9091 agent-bridge listen
# → espera el pedido del frontend
# → programas TODO lo que pidió (endpoints, modelo, BD, migraciones)
AGENT_BRIDGE=http://localhost:9091 agent-bridge respond frontend "✅ Endpoint listo en puerto 3000... cómo consumirlo..."

No le pedís al frontend que toque nada del backend.

Quick Start

1. Correr los bridges

Abrí dos terminales:

Terminal 1 — Frontend (hostea NATS):

agent-bridge serve -config ~/.agent-bridge/frontend.yaml

Terminal 2 — Backend:

export AGENT_BRIDGE=http://localhost:9091
agent-bridge serve -config ~/.agent-bridge/backend.yaml

2. Probar comunicación

Frontend AI pregunta algo:

agent-bridge ask --wait backend "Endpoint GET /api/hello"

Backend AI escucha, programa y responde:

export AGENT_BRIDGE=http://localhost:9091
agent-bridge listen   # recibe el mensaje al instante (WebSocket push)
agent-bridge respond frontend "Endpoint /api/hello listo en puerto 3000. curl http://localhost:3000/api/hello"
Nota: agent-bridge listen y agent-bridge ask --wait usan WebSocket push. Si el WebSocket no está disponible, caen automáticamente a polling cada 2 segundos. No necesitás hacer nada distinto.

Comandos CLI

ComandoDescripción
agent-bridge serve -config <archivo>Inicia el daemon del bridge
agent-bridge ask --wait <agente> <mensaje>Envía y espera respuesta (se bloquea hasta recibir, timeout 5 min)
agent-bridge listenEspera hasta que llegue un mensaje (WebSocket push + fallback polling, timeout 30 min)
agent-bridge respond <agente> <mensaje>Responde a un agente
agent-bridge checkRevisa mensajes nuevos (no bloqueante)
agent-bridge delegate --wait <agente> <tarea>Delega tarea y espera resultado
agent-bridge tasksLista tareas pendientes
agent-bridge watchMonitoreo en vivo de mensajes
agent-bridge initGenera archivo de configuración

Variable de entorno AGENT_BRIDGE (default: http://localhost:9090):

# Frontend (default)
AGENT_BRIDGE=http://localhost:9090

# Backend (necesario)
export AGENT_BRIDGE=http://localhost:9091

Flujo Frontend / Backend

Frontend pide un endpoint

1. Tu opencode ejecuta:

agent-bridge ask --wait backend "Crear endpoint GET /api/analytics/dashboard con activeUsers, dailyConversions, weeklyRevenue. Dame código completo y cómo consumirlo."

2. opencode se bloquea esperando respuesta ⏳ esperando respuesta de backend...

3. La IA del backend (Claude Code) corre:

AGENT_BRIDGE=http://localhost:9091 agent-bridge listen
# Recibe el mensaje al instante via WebSocket

4. Claude Code programa TODO: endpoints, modelo, lógica, migración. Y responde:

agent-bridge respond frontend "✅ Endpoint /api/analytics/dashboard listo en puerto 3000

📁 server.js líneas 22-48:
- GET /api/analytics/dashboard
- Modelo Analytics con activeUsers, dailyConversions, weeklyRevenue
- Base de datos SQLite con migración automática

🧪 curl http://localhost:3000/api/analytics/dashboard

📞 Consumir desde JS:
fetch('http://localhost:3000/api/analytics/dashboard')
  .then(r => r.json())
  .then(data => console.log(data))"

5. opencode recibe la respuesta y vos implementás la UI.

Importante: El backend programa TODO el backend. El frontend SOLO implementa la UI. Nadie toca lo del otro.

nats_port: 4222

mcp_server: false

listen_addr: "127.0.0.1:9091"

nats_port: 4222
db_path: "agent-bridge-backend.db"
server_mode: false
mcp_server: true
on_message: "opencode --take-order 'Mensaje de {{from}}: {{content}}'"

Conexión remota

Para conectar PCs en distintas redes (países distintos), la opción más simple es Tailscale o ZeroTier:

  1. Ambos instalan Tailscale (gratuito) o ZeroTier
  2. Se conectan a la misma red virtual
  3. Cada uno obtiene una IP tipo 100.x.x.x (Tailscale) o 10.x.x.x (ZeroTier)
  4. El que hostea NATS cambia su nats_url a su IP virtual
  5. El otro apunta su nats_url a la IP del que hostea
# En el config del que hostea NATS:
nats_url: "nats://100.x.x.x:4222"   # Su IP Tailscale

Alternativas

  • Tailscale — WireGuard, 0 config, gratuito. Recomendado.
  • ZeroTier — Similar, funciona en Windows/Linux/macOS.
  • VPS con NATS — Un servidor de $4/mes con NATS server. Ambos se conectan a la misma IP.

API: Mensajes

Enviar mensaje

POST /messages/send

curl -s localhost:9090/messages/send \
  -H 'Content-Type: application/json' \
  -d '{"to":"backend","type":"ask","content":"Endpoint GET /api/users"}'
CampoTipoRequeridoDescripción
tostring✅Identidad del agente destino
typestring✅ask, respond, task_delegate, task_result, share_context
contentstring✅Contenido del mensaje

Leer mensajes

GET /messages/new?unread=true

Marcar como leído

POST /messages/read

curl -s localhost:9090/messages/read \
  -H 'Content-Type: application/json' \
  -d '{"id":"uuid-del-mensaje"}'

API: Tareas

Delegar tarea

POST /tasks/delegate

curl -s localhost:9090/tasks/delegate \
  -H 'Content-Type: application/json' \
  -d '{"to":"backend","description":"Crear endpoint GET /api/analytics/dashboard con usuarios activos, conversiones y revenue semanal"}'

Listar tareas

GET /tasks/list?active=true

Actualizar estado

POST /tasks/status

curl -s localhost:9091/tasks/status \
  -H 'Content-Type: application/json' \
  -d '{"task_id":"uuid","status":"done","result":"Endpoint creado"}'

API: Compartir contexto

POST /context/share

curl -s localhost:9090/context/share \
  -H 'Content-Type: application/json' \
  -d '{"to":"backend","content":"Types que uso","files":["src/types/user.ts"]}'

API: Health

GET /health

curl localhost:9090/health
{"identity":"frontend","status":"ok"}

API: WebSocket

WS /ws

El endpoint WebSocket permite recibir mensajes en tiempo real. Cuando un mensaje llega al bridge via NATS, se envía automáticamente a todos los clientes WebSocket conectados. 

# Conectar via WebSocket (ejemplo con wscat)
wscat -c ws://localhost:9090/ws

# Al enviar un mensaje desde otro bridge, este cliente lo recibe al instante:
{"id":"...","from":"backend","to":"frontend","type":"respond","content":"Endpoint listo"}

Los comandos agent-bridge listen y agent-bridge ask --wait usan este endpoint automáticamente. Si falla, caen a polling.

MCP Tools (para Claude Code / Cursor)

Cuando mcp_server: true, el bridge expone un servidor MCP via stdio. Claude Code y Cursor lo detectan automáticamente.

ToolDescripción
agent_check_messagesRevisa mensajes nuevos
agent_send_messageEnvía un mensaje a otro agente
agent_delegate_taskDelega una tarea
agent_task_resultReporta resultado de tarea
agent_share_contextComparte contexto/archivos

Configurar en Claude Code

{
  "mcpServers": {
    "agent-bridge": {
      "command": "agent-bridge",
      "args": ["-config", "~/.agent-bridge/backend.yaml"],
      "env": {}
    }
  }
}
