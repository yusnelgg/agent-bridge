AgentBridge

Conecta agentes de IA entre distintas PCs para que colaboren y compartan informaciГіn en tiempo real.

v1.0.0 вҖў Go + NATS + MCP

IntroducciГіn

Agent Bridge es un daemon en Go que permite que agentes de IA (opencode, Claude Code, Cursor, cualquier AI que pueda hacer HTTP) se comuniquen entre sГӯ a travГ©s de distintas PCs. Es agnГіstico al modelo de IA: cada PC puede usar el asistente que quiera.

Caso de uso tГӯpico: Dos desarrolladores en distintos paГӯses вҖ” uno frontend, uno backend вҖ” quieren que sus asistentes AI colaboren. El frontend le pide a su IA un endpoint, su IA le pregunta a la IA del backend, la IA del backend lo implementa y reporta el resultado.

CaracterГӯsticas

  • Sin dependencias externas вҖ” NATS server embebido, no necesitas instalar nada
  • IA AgnГіstico вҖ” Cada lado usa el AI que quiera: opencode, Claude Code, Cursor, etc.
  • API REST local вҖ” Tu AI habla con el bridge via HTTP en localhost
  • MCP Server вҖ” Claude Code y Cursor se conectan nativamente via MCP
  • Persistencia SQLite вҖ” Mensajes y tareas se guardan localmente
  • P2P via NATS вҖ” ComunicaciГіn eficiente pub/sub entre bridges
  • Tailscale friendly вҖ” Funciona perfecto sobre Tailscale sin abrir puertos

El bridge de cada PC expone una API HTTP local. Cuando un agente AI quiere comunicarse con otro, envГӯa un mensaje a su bridge local, que lo publica en NATS. El bridge destino lo recibe y lo persiste en SQLite. El AI destino lo consulta via HTTP o lo recibe via MCP.

Quick Start

Prerrequisitos

  • Go 1.21+
  • Opcional: Tailscale para conectar PCs en distintas redes

1. Compilar

git clone <tu-repo> agent-bridge
cd agent-bridge
make build

2. Configurar

Crea un archivo agent-config.yaml:

identity: "frontend"
listen_addr: "127.0.0.1:9090"
nats_url: "nats://127.0.0.1:4222"
nats_port: 4222
db_path: "agent-bridge.db"
server_mode: true
mcp_server: false

3. Correr el bridge

./agent-bridge -config agent-config.yaml
Tip: Si server_mode es true, el bridge inicia su propio NATS server embebido. No necesitas instalar nada aparte. Solo un bridge necesita tener server_mode: true; los otros se conectan a Г©l.

4. Probar comunicaciГіn

# Enviar mensaje
curl -s localhost:9090/messages/send \
  -H 'Content-Type: application/json' \
  -d '{"to":"backend","type":"ask","content":"Hola backend"}'

# Leer mensajes recibidos
curl -s localhost:9090/messages/new?unread=true

MCP Tools (para Claude Code / Cursor)

Cuando mcp_server: true, el bridge expone un servidor MCP via stdio. Claude Code y Cursor lo detectan automГЎticamente. Las herramientas disponibles son:

ToolDescripciГіn
agent_check_messages Revisa mensajes nuevos de otros agentes
agent_send_message EnvГӯa un mensaje a otro agente
agent_delegate_task Delega una tarea a otro agente
agent_task_result Reporta el resultado de una tarea delegada

Flujo de trabajo

Frontend necesita un endpoint

1. TГә le dices a opencode:

2. opencode ejecuta:

IntegraciГіn con opencode

opencode puede ejecutar comandos shell. La forma mГЎs simple es que le digas: 

Ejecuta: curl -s localhost:9090/messages/send -H 'Content-Type: application/json' \
  -d '{"to":"backend","type":"ask","content":"..."}'

O crea un script agent.sh en tu PATH para hacerlo mГЎs limpio:

#!/bin/bash
# agent.sh вҖ” CLI para agent-bridge
BRIDGE="http://localhost:9090"

case "$1" in
  ask)
    curl -s $BRIDGE/messages/send \
      -H 'Content-Type: application/json' \
      -d "{\"to\":\"$2\",\"type\":\"ask\",\"content\":\"$3\"}"
    ;;
  check)
    curl -s $BRIDGE/messages/new?unread=true
    ;;
  delegate)
    shift
    curl -s $BRIDGE/tasks/delegate \
      -H 'Content-Type: application/json' \
      -d "{\"to\":\"$1\",\"description\":\"$2\"}"
    ;;
  tasks)
    curl -s $BRIDGE/tasks/list?active=${2:-true}
    ;;
  *)
    echo "Uso: agent ask|check|delegate|tasks"
    ;;
esac
Consejo: Pon agent.sh en tu PATH y desde opencode solo dices "Ejecuta: agent ask backend 'Necesito endpoint X'". Mucho mГЎs limpio.

ConexiГіn remota con Tailscale

Para conectar PCs en distintas redes (paГӯses distintos), Tailscale es la opciГіn mГЎs simple:

  1. Ambos instalan Tailscale
  2. Se conectan a la misma red (se crea una "tailnet")
  3. Cada uno obtiene una IP tipo 100.x.x.x
  4. El que hostea NATS usa IP 0.0.0.0:4222 (ya configurado)
  5. El otro apunta su nats_url a nats://100.x.x.x:4222

Razones para usar Tailscale

  • вң… No necesitas abrir puertos en el router
  • вң… No necesitas IP pГәblica
  • вң… ConexiГіn cifrada (WireGuard)
  • вң… Funciona detrГЎs de NAT, firewalls, etc.
  • вң… Gratis para uso personal
